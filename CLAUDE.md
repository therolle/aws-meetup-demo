# CLAUDE.md

## Project Overview

AWS Amplify Gen2 Todo demo app — React + TypeScript frontend with Amplify Gen2 backend (Cognito auth + DynamoDB-backed GraphQL API). Used as a demonstration of spec-driven agentic engineering at AWS Meetup Aarhus.

## Commands

- `npm run dev` — start Vite dev server
- `npm run build` — typecheck with `tsc` then build with Vite (output in `dist/`)
- `npm run preview` — preview production build locally
- `npm test` — run Vitest test suite
- `npx ampx sandbox` — start Amplify cloud sandbox for backend development
- `npx ampx sandbox delete` — tear down sandbox resources

## Architecture

**Frontend** (`src/`): React 18 + Vite 6 + Tailwind CSS v4 (via `@tailwindcss/vite` plugin). The `Authenticator` component wraps the app in `main.tsx`, so all routes require sign-in. `App.tsx` uses `generateClient<Schema>()` for typed data operations and `observeQuery()` for real-time subscription.

**Backend** (`amplify/`): Amplify Gen2 code-first backend defined in TypeScript.
- `backend.ts` — entry point, wires auth + data resources
- `auth/resource.ts` — Cognito auth with email login
- `data/resource.ts` — GraphQL schema with `Todo` model (owner-authorized, DynamoDB)

**Data flow**: The typed `Schema` export from `amplify/data/resource.ts` is imported by the frontend for type-safe client generation. `amplify_outputs.json` (gitignored) provides runtime configuration.

## Conventions

### Code Standards
- **Conventional commits**: `feat:`, `fix:`, `docs:`, `refactor:`, `chore:`
- **TypeScript strict mode** — no `any`, no `as` casts without justification
- **Tailwind v4 tokens only** — no inline hex colors, use `@theme` variables in `src/index.css`
- **All components must be accessible** — ARIA labels, keyboard navigation, focus management
- **Named exports** for components, default export only for `App.tsx`

### Package Manager
- npm (lockfile: `package-lock.json`) — never use yarn/pnpm/bun

### Amplify Conventions
- Always use typed `generateClient<Schema>()` — never untyped client
- Authorization: owner-based by default, explicit justification for public access
- Schema changes require sandbox testing before commit
- Backend resources defined code-first in `amplify/` directory

### CI/CD
- `amplify.yml` — backend deploys with `ampx pipeline-deploy`, frontend builds with `npm run build`

## Test-Driven Development (NON-NEGOTIABLE)

**Every feature, bug fix, and refactor starts with a failing test.** No production code without a test that demands it.

### The Cycle: RED → GREEN → REFACTOR

1. **RED**: Write a failing test that describes desired behavior
   - Test must fail for the right reason (not a syntax error)
   - Test describes WHAT, not HOW — test behavior, not implementation
2. **GREEN**: Write the minimum code to make the test pass
   - Only enough code to satisfy the failing test — nothing more
3. **REFACTOR**: Improve code while keeping tests green
   - Only if it adds value — don't refactor for the sake of it

### Rules

- **Tests live next to source files**: `ComponentName.test.tsx` beside `ComponentName.tsx`
- **Use React Testing Library** — query by role, label, text (not test IDs or CSS selectors)
- **User-centric tests** — test what the user sees and does, not component internals
- **No mocking the thing being tested** — only mock external dependencies
- **Run `npm test` after every RED→GREEN cycle** to verify

### Test Naming Convention

```typescript
describe("ComponentName", () => {
  it("renders the expected initial state", () => { ... });
  it("does X when user does Y", () => { ... });
});
```

### Commit Progression

```
test: add failing test for todo completion toggle
feat: implement todo completion toggle
refactor: extract toggle handler for clarity
```

## Quality Gates

- `npm run build` must pass before any commit (TypeScript + Vite build)
- `npm test` must pass before any commit
- No direct DOM manipulation — use React state and refs
- Amplify backend changes require `npx ampx sandbox` testing first
- All new components must have corresponding test files
- **TDD evidence required** — every feature commit must be preceded by a test commit

## What NOT to Do

- **Never commit** `amplify_outputs.json` — it's gitignored, generated by sandbox
- **Never hardcode** AWS credentials or secrets
- **Never bypass** TypeScript errors with `@ts-ignore` or `@ts-expect-error`
- **Never use inline styles** for colors — use Tailwind design tokens
- **Never skip** the Authenticator wrapper — all routes require auth
- **Never use `var`** — always `const` or `let`
- **Never mutate state directly** — use React setState patterns

## Spec-Driven Development

This project uses Kiro for spec-driven development:
- Requirements, design, and tasks live in `.kiro/specs/`
- Every feature starts with a spec triple: `requirements.md` → `design.md` → `tasks.md`
- Steering documents in `.kiro/steering/` define product, structure, and tech conventions

Agent specifications live in `.claude/agents/` — these are contracts, not prompts.
